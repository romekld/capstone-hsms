## Phase Boundary

Build the complete API layer, service logic, repositories, and frontend UI for prenatal enrollment/visits/overdue/high-risk and postpartum delivery recording — on top of the ORM models and Pydantic schemas already delivered in Phase 4.1.

**Includes:**
- `PrenatalRepository`, `PostpartumRepository` (repositories/prenatal.py, repositories/postpartum.py)
- `PrenatalService`, `PostpartumService` (services/prenatal.py, services/postpartum.py)
- `routers/prenatal.py` + `routers/postpartum.py` (separate files, registered under main app)
- Phase 4.2 Alembic migration: adds delivery columns to `prenatal_enrollments` + `fp_method_chosen` to `postpartum_visits`
- Frontend Step 1: Prenatal tab + Postpartum tab on patient profile (individual patient view)
- Frontend Step 2: `/maternal/prenatal` and `/maternal/postpartum` as **Digital TCL pages** (wide ledger-style tables mirroring the DOH paper TCL format)
- Sidebar nav badges for overdue counts

**Build order (two steps):**
1. **Backend + Individual Profile tabs** — All repositories, services, routers, and the Prenatal/Postpartum tabs on the patient profile page. Nurses can enroll patients and record visits before the TCL pages exist.
2. **Digital TCL pages** — `/maternal/prenatal` and `/maternal/postpartum` as wide program-level list views. Built after backend is solid so the TCL can rely on a rich, JOIN-enabled enrollment+visits endpoint.

**Out of scope:** EPI (Phase 4.3), Nutrition (Phase 4.4), TB/NCD (Phase 5), FHSIS reporting (Phase 8).

<decisions>
## Implementation Decisions

### API Route Structure
- **Patient-scoped operations** use nested routes under `/patients/{id}/`:
  ```
  POST /patients/{id}/prenatal/enrollment      ← enroll
  GET  /patients/{id}/prenatal/enrollment      ← get enrollment
  POST /patients/{id}/prenatal/visits          ← add visit
  GET  /patients/{id}/prenatal/visits          ← list visits
  POST /patients/{id}/prenatal/delivery        ← record delivery (closes prenatal, creates postpartum)

  GET  /patients/{id}/postpartum/enrollment    ← get enrollment (auto-created by delivery)
  POST /patients/{id}/postpartum/visits        ← add visit
  GET  /patients/{id}/postpartum/visits        ← list visits
  ```
- **Program list operations** use flat routes that power the Digital TCL pages:
  ```
  GET /prenatal/enrollments                    ← all active enrollments + visits (BHS-scoped) — primary TCL endpoint
  GET /prenatal/enrollments?overdue=true       ← filter: overdue only
  GET /prenatal/enrollments?high_risk=true     ← filter: high-risk only
  GET /postpartum/enrollments                  ← all postpartum enrollments + visits (BHS-scoped)
  GET /postpartum/enrollments?overdue=true     ← filter: any overdue slot
  ```
  The primary TCL endpoint returns enrollments **with their visits embedded** (joined response) — the frontend does not need to make per-patient follow-up calls to render the wide table.
- **Router organization:** Separate `routers/prenatal.py` + `routers/postpartum.py` — keeps each program self-contained; consistent with EPI/Nutrition routers coming in 4.3/4.4
- **Role access:** nurse/midwife write, physician read (view enrollment + visits), CHO/DSO/PHIS coordinator read-only city-wide — mirrors Phase 3 consultation RBAC exactly

### Prenatal Enrollment Closure (Delivery Status)
- **Phase 4.2 migration** adds three columns to `prenatal_enrollments`:
  - `delivered_at TIMESTAMPTZ` — nullable; NULL = active enrollment, non-NULL = delivered
  - `delivery_outcome TEXT` — `'LIVE_BIRTH'` | `'STILLBIRTH'` | `'ABORTION'`
  - `delivery_place TEXT` — free text, nullable (BHC name, hospital, etc.)
- **Active enrollment query:** `WHERE delivered_at IS NULL AND deleted_at IS NULL`
- **Record Delivery endpoint:** `POST /patients/{id}/prenatal/delivery` — dedicated action endpoint (not a PATCH); atomically sets `delivered_at`/`delivery_outcome`/`delivery_place` and auto-creates the postpartum enrollment
- **Delivery response:** Returns a combined object with both the updated prenatal enrollment (now closed) and the newly created postpartum enrollment (with `day1_date`, `week1_date`, `week6_date`). Frontend updates both tabs without an extra API call.

### AOG-Based next_visit_date Calculation (PRNT-03)
- **DOH-standard frequency:**
  - AOG < 28 weeks → `next_visit_date = visit_date + 4 weeks`
  - AOG 28–35 weeks → `next_visit_date = visit_date + 2 weeks`
  - AOG ≥ 36 weeks → `next_visit_date = visit_date + 1 week`
  - AOG is NULL on a visit record → default to 4 weeks
- **Capped at EDC:** If computed `next_visit_date > edc`, service sets `next_visit_date = edc`
- **Initial scheduling:** At enrollment creation, service computes current AOG from `lmp` (today's date − LMP in weeks), applies the same frequency rule, and sets `next_visit_date` immediately — never NULL after enrollment

### High-Risk Flag (PRNT-05, PRNT-06)
- **Auto-flag triggers** (service evaluates after each visit save): BP ≥ 140/90, Hgb < 11 g/dL, para ≥ 5 (checked at enrollment), GDM positive, nurse manual checkbox
- **is_high_risk never auto-resets** — once flagged (by auto-trigger or manual), only the nurse can unset it by editing the enrollment. Service auto-trigger can only SET to `true`, never reset to `false`
- **Manual flag reason:** Stored in `risk_factors TEXT` (existing enrollment column) — no new column needed. Auto-flag reason is implicit from the visit data (BP/Hgb values visible in visit history)

### Postpartum Visit Fields (PNPL-03)
- **Phase 4.2 migration** adds one column to `postpartum_visits`:
  - `fp_method_chosen TEXT` — optional; records FP method if nurse provides counseling and patient makes a choice
- **Pydantic validation** on existing TEXT columns (same pattern as EPI vaccine — validate at API layer, keep DB column as TEXT):
  - `wound_status: Literal['GOOD', 'SUBINVOLUTED', 'BOGGY'] | None`
  - `newborn_status: Literal['ALIVE_WELL', 'DECEASED', 'REFERRED'] | None`
- **Postpartum overdue detection:** `postpartum_visits.visit_type: Literal["day1", "week1", "week6"]` already exists. Overdue = no visit recorded with that type AND corresponding scheduled date (`day1_date`/`week1_date`/`week6_date`) < `CURRENT_DATE`

### Digital TCL Pages (Program-Level List Views)

**Concept:** `/maternal/prenatal` and `/maternal/postpartum` are Digital TCL (Target Client List) pages — wide, ledger-style tables that mirror the DOH paper TCL format. One row = one enrolled patient. Visit data spreads across columns. Nurses see the same clinical picture they would on paper, but live and filterable.

**Prenatal TCL (`/maternal/prenatal`) columns:**
```
Name | LMP | EDC | G | P | V1 Date | V1 AOG | V1 BP | V1 Wt | V2 Date | V2 AOG | V2 BP | V2 Wt | ... | Delivery | Outcome
```
- Number of visit columns expands dynamically based on the maximum visit count in the dataset (or fixed at the expected DOH visit count — Claude's discretion on whether to cap or expand)
- Overdue visit cells highlighted red (the expected-visit-date cell is past due with no recorded date)
- HIGH RISK badge on the patient name cell (row-level indicator, not a separate section)
- Horizontal scroll expected (~20–40 columns)

**Postpartum TCL (`/maternal/postpartum`) columns:**
```
Name | Delivery | Outcome | Day1 Date | Day1 BP | Wk1 Date | Wk1 BP | Wk6 Date | Wk6 BP | FP Method
```
- ~10–12 columns; fixed slot structure (no dynamic column expansion needed)
- Overdue slots highlighted red; completed slots shown normally

**Scoping:**
- Nurse sees own BHS patients only by default
- CHO/DSO/PHIS coordinator sees city-wide with a BHS filter dropdown (same `CROSS_BHS_ROLES` pattern from Phase 3)

**Badge refresh strategy:** React Query `invalidateQueries` on visit-record mutations — no polling. Badge re-fetches immediately after a nurse records a visit.

**Navigation from TCL row click:** Navigates to `/patients/{id}?tab=prenatal` (or `?tab=postpartum`). `PatientProfilePage` reads the `tab` query param to pre-select the active tab.

### Claude's Discretion
- Exact SQL for AOG-based overdue detection query (join strategy between enrollments + visits)
- Postpartum overdue query joining enrollment + visits by visit_type
- Whether the `/prenatal/enrollments` flat list routes live in a separate router file (e.g., `routers/maternal.py`) or as additional endpoints in `routers/prenatal.py`
- Exact loading states and skeleton design for program tabs
- Error message wording for "patient already enrolled in active prenatal care"

</decisions>

<specifics>
## Specific Ideas

- **Digital TCL concept:** The `/maternal/prenatal` and `/maternal/postpartum` pages are "Digital TCLs" — wide tables that mirror the DOH paper Target Client List ledger format. Nurses already know this format; the goal is to replicate it digitally with live data and visual overdue indicators.
- **Two-step build order** (explicit for planners): Build backend + profile tabs first, then TCL pages. This ensures all API endpoints are solid before the complex wide-table frontend is built on top of them.
- `POST /patients/{id}/prenatal/delivery` is an **action endpoint**, not a REST resource update — name reflects the clinical action rather than the data being patched
- Delivery response returns a combined shape: `{ prenatal: PrenatalEnrollmentResponse, postpartum: PostpartumEnrollmentResponse }` — frontend can update both tabs in a single mutation callback
- `next_visit_date` is computed at enrollment creation (not left NULL) so overdue detection has a baseline from day one
- High-risk is sticky — auto-flag can set it, only human unsets it. This prevents automatic clearing of a flag a nurse set for a reason not captured by the auto-check rules
- `wound_status` validation: `GOOD` / `SUBINVOLUTED` / `BOGGY` — these are standard BHS terminology for uterine involution assessment
- Postpartum overdue uses `visit_type` to match each scheduled slot exactly: a `day1` visit only covers the Day 1 slot, not the Week 1 slot
- TCL endpoint (`GET /prenatal/enrollments`) returns enrollments with visits embedded — the frontend renders the wide table from a single response without per-row follow-up calls

</specifics>
