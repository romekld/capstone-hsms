## Summary

Phase 4.2 builds the complete service layer (repositories, services, routers) for prenatal and postpartum programs on top of the ORM models and Pydantic schemas already delivered in Phase 4.1. The work divides cleanly into two steps: backend + patient profile tabs first, then Digital TCL pages.

The critical new work in this phase is business logic density: EDC auto-calculation from LMP, AOG-based next visit scheduling, high-risk auto-flagging at two evaluation points (enrollment creation and post-visit), and the atomic delivery transition that closes prenatal and opens postpartum. All of these are service-layer concerns — the repository layer remains thin (query + flush).

The Digital TCL pages are the frontend challenge: wide ledger-style tables with up to 10+ columns, sticky first column, overdue cell highlighting, and embedded visit data from a single API response. This requires careful use of CSS `position: sticky` and the `overflow-x: auto` container pattern, which are distinct from the standard table patterns used in the existing patient list and consultation list.

**Primary recommendation:** Follow the established `ConsultationRepository → PatientService → patient router` layering exactly. The only new patterns are the delivery action endpoint (returns a combined shape), the TCL list endpoints with JOIN-embedded visits, and the wide sticky-column table frontend.

### Pattern 1: Program Repository (extends BaseRepository)

Follows `ConsultationRepository` exactly. The key constraint: `PrenatalEnrollment` has `health_station_id`, so `_isolation_filter()` from `BaseRepository` applies directly for enrollment-level queries. For visit-level queries, JOIN through the enrollment.

```python
# Source: backend/app/repositories/consultation.py (established pattern)
from sqlalchemy import select
from sqlalchemy.orm import selectinload
from app.models.prenatal import PrenatalEnrollment, PrenatalVisit
from app.repositories.base import BaseRepository

class PrenatalRepository(BaseRepository):

    async def get_active_enrollment(self, patient_id: int) -> PrenatalEnrollment | None:
        stmt = (
            select(PrenatalEnrollment)
            .where(
                PrenatalEnrollment.patient_id == patient_id,
                PrenatalEnrollment.delivered_at.is_(None),
                PrenatalEnrollment.deleted_at.is_(None),
            )
        )
        result = await self.session.execute(stmt)
        return result.scalar_one_or_none()

    async def create_enrollment(self, **kwargs) -> PrenatalEnrollment:
        enrollment = PrenatalEnrollment(**kwargs)
        self.session.add(enrollment)
        await self.session.flush()
        return enrollment

    async def create_visit(self, **kwargs) -> PrenatalVisit:
        visit = PrenatalVisit(**kwargs)
        self.session.add(visit)
        await self.session.flush()
        return visit

    async def list_visits(self, enrollment_id: int) -> list[PrenatalVisit]:
        stmt = (
            select(PrenatalVisit)
            .where(
                PrenatalVisit.prenatal_enrollment_id == enrollment_id,
                PrenatalVisit.deleted_at.is_(None),
            )
            .order_by(PrenatalVisit.visit_date.asc())
        )
        result = await self.session.execute(stmt)
        return list(result.scalars().all())
```

### Pattern 2: TCL List Query with Embedded Visits

The `GET /prenatal/enrollments` endpoint must return enrollments with their visits embedded in a single query. Use `selectinload` for the visits relationship to avoid N+1.

```python
# Source: established SQLAlchemy 2.0 async selectinload pattern
from sqlalchemy import select
from sqlalchemy.orm import selectinload

async def list_active_enrollments_with_visits(
    self, status: str = "active"
) -> list[PrenatalEnrollment]:
    stmt = select(PrenatalEnrollment).options(
        selectinload(PrenatalEnrollment.visits),
        selectinload(PrenatalEnrollment.patient),
    )
    # Apply BHS isolation (from BaseRepository._isolation_filter)
    stmt = self._isolation_filter(stmt, PrenatalEnrollment)

    if status == "active":
        stmt = stmt.where(
            PrenatalEnrollment.delivered_at.is_(None),
            PrenatalEnrollment.deleted_at.is_(None),
        )
    elif status == "delivered":
        stmt = stmt.where(
            PrenatalEnrollment.delivered_at.isnot(None),
            PrenatalEnrollment.deleted_at.is_(None),
        )
    else:  # "all"
        stmt = stmt.where(PrenatalEnrollment.deleted_at.is_(None))

    stmt = stmt.order_by(PrenatalEnrollment.created_at.desc())
    result = await self.session.execute(stmt)
    return list(result.scalars().unique().all())
```

**Critical:** Use `.unique()` after `.scalars()` when using `selectinload` to prevent duplicate rows from the JOIN — SQLAlchemy 2.0 requires this.

### Pattern 3: EDC + AOG Service Logic

```python
# Source: CONTEXT.md locked decisions
from datetime import date, timedelta

def _compute_edc(lmp: date) -> date:
    return lmp + timedelta(days=280)

def _compute_aog_weeks(lmp: date, reference_date: date) -> int:
    delta = reference_date - lmp
    return delta.days // 7

def _compute_next_visit_date(
    visit_date: date, aog_weeks: int | None, edc: date
) -> date:
    aog = aog_weeks if aog_weeks is not None else 0  # treat None as <28
    if aog >= 36:
        next_date = visit_date + timedelta(weeks=1)
    elif aog >= 28:
        next_date = visit_date + timedelta(weeks=2)
    else:
        next_date = visit_date + timedelta(weeks=4)
    # Cap at EDC
    return min(next_date, edc)
```

### Pattern 4: High-Risk Evaluation

```python
# Source: CONTEXT.md locked decisions
from datetime import date

def _evaluate_high_risk(
    enrollment,
    patient_birthdate: date,
    reference_date: date,
    visit=None,
) -> bool:
    # Never auto-reset — return True if ALREADY flagged
    if enrollment.is_high_risk:
        return True

    # Enrollment-level triggers
    if enrollment.para >= 5:
        return True
    age_years = (reference_date - patient_birthdate).days // 365
    if age_years < 18 or age_years > 35:
        return True

    # Visit-level triggers (only when a visit is provided)
    if visit is not None:
        if (
            visit.bp_systolic is not None
            and visit.bp_diastolic is not None
            and (visit.bp_systolic >= 140 or visit.bp_diastolic >= 90)
        ):
            return True
        if visit.hgb is not None and visit.hgb < 11:
            return True
        if visit.gdm_positive is True:
            return True

    return False
```

### Pattern 5: Delivery Action Endpoint (Atomic)

The delivery endpoint must atomically close prenatal and create postpartum within the same SQLAlchemy session (same transaction). Use `session.flush()` between the two operations — the `session.commit()` at the end commits both together.

```python
# Source: CONTEXT.md decisions — action endpoint pattern
async def record_delivery(
    self, patient_id: int, body: DeliveryCreate
) -> DeliveryResponse:
    enrollment = await self.prenatal_repo.get_active_enrollment(patient_id)
    if not enrollment:
        raise HTTPException(404, "No active prenatal enrollment found.")

    # Step 1: Close the prenatal enrollment
    enrollment.delivered_at = body.delivery_date
    enrollment.delivery_outcome = body.delivery_outcome
    enrollment.delivery_place = body.delivery_place
    await self.session.flush()

    # Step 2: Create postpartum enrollment (shared method)
    pp_enrollment = await self.postpartum_service._create_postpartum_enrollment(
        patient_id=patient_id,
        delivery_date=body.delivery_date,
        delivery_outcome=body.delivery_outcome,
        delivery_place=body.delivery_place,
        prenatal_enrollment_id=enrollment.id,
    )

    await self.session.commit()
    # ... build combined response
```

### Pattern 7: Postpartum Overdue Detection Query

Overdue = a scheduled visit slot has no recorded visit AND the scheduled date is in the past.

```python
# Source: CONTEXT.md decisions — query-time computation, no cron flag
from sqlalchemy import select, func, case, exists
from app.models.postpartum import PostpartumEnrollment, PostpartumVisit

# Overdue count for sidebar badge — count enrollments where ANY slot is past due
# without a recorded visit for that slot type
async def count_overdue(self) -> int:
    today = date.today()

    # Subquery: does a day1 visit exist for this enrollment?
    day1_done = (
        select(func.count(PostpartumVisit.id))
        .where(
            PostpartumVisit.postpartum_enrollment_id == PostpartumEnrollment.id,
            PostpartumVisit.visit_type == "day1",
            PostpartumVisit.deleted_at.is_(None),
        )
        .correlate(PostpartumEnrollment)
        .scalar_subquery()
    )

    stmt = (
        select(func.count(PostpartumEnrollment.id))
        .where(
            PostpartumEnrollment.deleted_at.is_(None),
            # Any slot overdue: scheduled date < today AND no visit of that type
            (
                (PostpartumEnrollment.day1_date < today) & (day1_done == 0)
                | (PostpartumEnrollment.week1_date < today) & (week1_done == 0)
                | (PostpartumEnrollment.week6_date < today) & (week6_done == 0)
            ),
        )
    )
    stmt = self._isolation_filter(stmt, PostpartumEnrollment)
    result = await self.session.execute(stmt)
    return result.scalar_one()
```

**Alternative simpler approach for the TCL display** (mark individual cells overdue on the frontend rather than a DB overdue query): The TCL endpoint returns all enrollment + visit data; the frontend computes "is this slot overdue?" as `slot_date < today && no_visit_for_slot`. This is the recommended approach for the TCL page — no additional DB query needed.

### Anti-Patterns to Avoid

- **Putting `health_station_id` on visit tables.** Visit tables have no `health_station_id` by project design (Phase 4.1 locked decision). BHS isolation for visit queries MUST go through a JOIN to the enrollment table.
- **Hard-coding BHS-scoped roles in the prenatal/postpartum service.** Import `CROSS_BHS_ROLES` from `app.repositories.base` — the same constant used by all other services.
- **Double-wrapping `require_role()` in `Depends()`.** The `require_role()` function already returns `Depends(_guard)`. Pass it directly as `_=require_role(ROLES)` — see `routers/patient.py` line 46.
- **Calling `session.commit()` inside `flush()`.** Repositories flush, services commit. Mixing breaks the atomic delivery transaction.
- **Auto-resetting `is_high_risk` to False on post-visit evaluation.** The flag is sticky — auto-trigger can only set to True. The `_evaluate_high_risk()` function must return True if the enrollment is ALREADY high-risk.
- **Returning raw ORM objects from routers.** Always build the Pydantic response schema before returning. FastAPI does not guarantee `model_config = {"from_attributes": True}` is invoked unless the response_model is set.
- **Using `.all()` without `.unique()` after `selectinload`.** SQLAlchemy 2.0 returns duplicate rows when selectinload joins produce cartesian product at the execution level. Always call `result.scalars().unique().all()`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Sticky first column on horizontal-scroll table | Custom JS scroll sync | CSS `position: sticky; left: 0` on `<TableCell>` + `overflow-x: auto` on container | Native browser feature; JS scroll sync has lag and edge cases |
| BHS data isolation | Per-endpoint `WHERE health_station_id =` clauses | `BaseRepository._isolation_filter()` | Centralized; already handles CROSS_BHS_ROLES exception |
| "Is this patient overdue?" per-row calculation in frontend | Additional per-patient API calls | Embed all enrollment + visit data in the TCL list response; compute overdue client-side | Single response is sufficient; per-row calls would be N+1 |
| Duplicate active enrollment check | Application-level flag | Query-time check in service: `get_active_enrollment()` returns None → enroll; returns enrollment → raise 409 | Idempotent DB query is simpler and more reliable |
| Postpartum schedule date calculation | Date library | `timedelta(days=0/7/42)` from Python stdlib | All three intervals are fixed constants; no library needed |
| Audit logging for clinical create actions | Custom audit table per program | Re-use existing `_write_audit()` private method pattern from `PatientService` | Established project pattern; consistent audit trail |

**Key insight:** The biggest "don't hand-roll" trap in this phase is building a separate overdue detection cron job or Celery task for prenatal/postpartum. The CONTEXT.md decision is explicit: query-time computation (`WHERE next_visit_date < NOW()`) is sufficient for BHS volumes (< 100 active enrollments per BHS). No scheduler needed.

## Common Pitfalls

### Pitfall 1: `selectinload` Without `.unique()` Produces Duplicate Rows

**What goes wrong:** `result.scalars().all()` after a query with `selectinload` returns duplicate enrollment objects — one copy per related visit.
**Why it happens:** SQLAlchemy 2.0 executes a JOIN under the hood for selectinload; the scalar result contains duplicate parent rows before deduplication.
**How to avoid:** Always call `result.scalars().unique().all()` when the query uses `selectinload` for a one-to-many relationship.
**Warning signs:** TCL table shows the same patient row multiple times; count doesn't match DB count.

### Pitfall 2: `delivered_at` Column Missing from ORM Model at Phase Start

**What goes wrong:** `PrenatalEnrollment` in Phase 4.1 does NOT have `delivered_at`, `delivery_outcome`, or `delivery_place` columns. The Phase 4.2 migration adds them. If the migration hasn't run, any attempt to set `enrollment.delivered_at = ...` will fail with an `AttributeError` on the ORM object or a DB error.
**Why it happens:** Phase 4.2 must run migration `0006` BEFORE executing any delivery logic. The ORM model also needs to be updated to add the new mapped columns.
**How to avoid:** Wave 0 task in the plan: update `backend/app/models/prenatal.py` to add the three new columns AND create the `0006_prenatal_delivery_columns.py` migration file. Run `alembic upgrade head` before any endpoint test.

### Pitfall 3: AOG Calculation Uses the Wrong Reference Date

**What goes wrong:** At enrollment creation, `next_visit_date` must be computed from the CURRENT date (today), not from `lmp` alone. AOG in weeks = `(today - lmp).days // 7`. Using `lmp` as the reference gives AOG = 0, which always falls into the `< 28 weeks` bucket and sets `next_visit_date = today + 4 weeks` even for third-trimester enrollments.
**Why it happens:** Confusing "current AOG" (computed at enrollment time from today) with "visit AOG" (recorded by nurse on the visit form).
**How to avoid:** In `PrenatalService.enroll()`, always pass `reference_date=date.today()` to `_compute_aog_weeks()`. On visits, use the `aog_weeks` value the nurse enters (which may differ from the computed value if enrollment was delayed).

### Pitfall 4: High-Risk Flag Age Check Off-By-One (Year Boundary)

**What goes wrong:** Computing age as `(today - birthdate).days // 365` gives 17 on the day before a patient's 18th birthday and 17 the day before a patient's 18th birthday — correct. But it gives 17 for the entire year before the birthday, even in leap years. For a clinical compliance system this is acceptable, but ensure the formula is applied consistently.
**Why it happens:** Leap year calculation using integer division of days.
**How to avoid:** Use `(reference_date - birthdate).days // 365` consistently. For this domain (age < 18 or > 35), the 1-day error near a birthday is clinically irrelevant. Do not introduce `dateutil.relativedelta` just for this check.

### Pitfall 5: Trying to `selectinload` Through `lazy="raise"` Relationships

**What goes wrong:** The ORM models use `lazy="raise"` for all relationships (Phase 4.1 convention). Accessing `enrollment.visits` or `enrollment.patient` outside a `selectinload` context raises `sqlalchemy.exc.InvalidRequestError: 'lazy' loading is not allowed`.
**Why it happens:** `lazy="raise"` prevents accidental N+1 queries by raising immediately instead of silently executing a query. Every relationship access must be explicitly loaded via `selectinload` in the repository query.
**How to avoid:** Any repository method that needs related objects MUST add `.options(selectinload(...))` to the query. The `create()` methods return plain ORM objects — callers cannot access relationships on them without a separate fetch.
**Warning signs:** `InvalidRequestError` in tests or at runtime when accessing `.patient`, `.visits`, etc. on an ORM instance returned by `create()`.

### Pitfall 6: `?tab=` Deep-Link Breaks on Page Refresh Without `useSearchParams`

**What goes wrong:** TCL row click navigates to `/patients/{id}?tab=prenatal`. On the patient profile page, if `defaultValue` is hardcoded to `"consultations"`, the Prenatal tab is not shown on initial load even when `?tab=prenatal` is in the URL.
**Why it happens:** `<Tabs defaultValue="consultations">` ignores URL state.
**How to avoid:** Read `searchParams.get("tab")` in `PatientProfilePage` and use it as `defaultValue`. If absent, fall back to `"consultations"`.

### Pitfall 7: Sticky Column Background Shows Through Other Rows on Scroll

**What goes wrong:** The sticky first column appears transparent when rows scroll under it, showing row content bleeding through.
**Why it happens:** The sticky cell has no explicit background color; it inherits `transparent` from the table row.
**How to avoid:** Explicitly set `bg-card` (or `bg-background`) on every sticky cell — both `<TableHead>` and `<TableCell>` in the first column. This must be applied to EVERY row, not just the header.

### Pitfall 8: TCL Column Count Mismatch Between Visit Slots and Response Data

**What goes wrong:** The TCL renders 9 visit columns (V1–V9). If the API response returns visits ordered differently (newest first instead of oldest first), V1 shows the most recent visit, which is clinically misleading.
**Why it happens:** Default `order_by` in the repository uses `created_at.desc()` (newest first). For TCL, visits must be ordered `visit_date.asc()` (oldest first).
**How to avoid:** The `list_visits()` repository method used by the TCL endpoint MUST order by `visit_date ASC`. The V1 column always maps to the chronologically first visit.

### Pitfall 9: Standalone Postpartum Enrollment Overwrites Existing Enrollment

**What goes wrong:** A nurse enrolls a patient in postpartum care via the standalone path (`POST /patients/{id}/postpartum/enrollment`). If there is already an active postpartum enrollment (created via delivery), the new enrollment creates a duplicate.
**Why it happens:** No uniqueness constraint on `(patient_id, deleted_at IS NULL)` for postpartum enrollments.
**How to avoid:** `PostpartumService._create_postpartum_enrollment()` must check for an existing active enrollment (no `deleted_at`, no completion) and raise 409 if one exists, with message "Patient already has an active postpartum enrollment."

## Open Questions

1. **Router file organization for TCL list routes**
   - What we know: CONTEXT.md allows either `routers/prenatal.py` (all prenatal routes including list) or a separate `routers/maternal.py`
   - What's unclear: Whether keeping patient-scoped + list routes in the same file creates a naming confusion
   - Recommendation: Keep all prenatal routes in `routers/prenatal.py` with prefix `/patients` for scoped routes and no prefix for list routes. Declare both in the same `APIRouter` using different sub-paths. This matches the single-file pattern used for `routers/patient.py`.

2. **Sidebar overdue badge data source**
   - What we know: Sidebar must show live overdue counts
   - What's unclear: Whether the sidebar reads from a dedicated `GET /prenatal/overdue-count` endpoint or from the TCL list response length
   - Recommendation: Add lightweight `GET /prenatal/overdue-count` and `GET /postpartum/overdue-count` endpoints that return `{"count": N}`. The sidebar component calls these on mount and after mutations. Thin endpoint, no pagination, no visit data — just the count.

3. **"Register Newborn" post-delivery flow UI trigger**
   - What we know: After recording a live birth delivery, a dialog prompts "Register Newborn". The newborn is a new patient registered via `POST /patients`.
   - What's unclear: Whether this is a toast notification with a "Register" action button or a blocking modal.
   - Recommendation: Non-blocking toast with an action button (using `sonner` toast API — already in project). Blocking modal disrupts nurse flow if they need to dismiss it. The toast action navigates to `/patients/new` with query params pre-filling the mother's BHS and barangay.
