# Phase 4.1: MCH Shared Data Model - Context

**Gathered:** 2026-03-18 (updated 2026-03-19)
**Status:** Ready for planning

<domain>
## Phase Boundary

Create the ORM models, Pydantic schemas, and a **single Alembic migration** for all eight MCH tables needed by Phases 4.2–4.4. No API endpoints, no frontend UI — this phase delivers the schema foundation only.

**Tables:** `prenatal_enrollments`, `prenatal_visits`, `postpartum_enrollments`, `postpartum_visits`, `epi_enrollments`, `epi_vaccinations`, `nutrition_enrollments`, `nutrition_visits`

**Out of scope:** All API routes, services, repositories, and frontend work (those belong to Phases 4.2, 4.3, and 4.4).

</domain>

<decisions>
## Implementation Decisions

### ORM models
- All 8 models use `TimestampMixin` + `SoftDeleteMixin` from `backend/app/core/base.py`
- All enrollment tables: `patient_id FK → patients.id` + `health_station_id FK → health_stations.id`
- All relationships use `lazy="raise"` — explicit joins only
- All 8 tables carry `recorded_by FK → users.id` (mirrors the `consultations` pattern established in Phase 3)

### Barangay isolation pattern
- `health_station_id` is on **enrollment tables only** — visit tables do NOT carry it
- Repository isolation for visit queries JOINs through the enrollment table to reach `health_station_id`
- This avoids denormalizing BHS identity across 4 extra tables

### Offline sync fields (all 8 tables)
- All 8 MCH tables carry `local_id UUID` + `status record_status` columns
- `record_status` is a **shared PostgreSQL ENUM** (defined once in this migration, reused by future TB/NCD/SURV tables)
- Values: `PENDING` (local only), `SYNCED` (confirmed by server), `CONFLICT` (requires nurse review)
- Default: `PENDING`

### Postpartum → Prenatal linkage
- `postpartum_enrollments` carries `prenatal_enrollment_id FK → prenatal_enrollments.id`
- FK is **nullable** — some postpartum records have no matching prenatal enrollment in the system (deliveries at external facilities)
- `ON DELETE SET NULL` — if prenatal enrollment is soft-deleted, the postpartum record remains with `prenatal_enrollment_id = NULL`

### Next visit date placement
- **Prenatal:** `prenatal_enrollments.next_visit_date DATE` — updated by service layer after each prenatal visit save; overdue detection queries the enrollment table directly
- **Postpartum:** `postpartum_enrollments` carries three fixed schedule columns: `day1_date DATE`, `week1_date DATE`, `week6_date DATE` — all computed from `delivery_date` at enrollment creation; overdue detection checks each column against `CURRENT_DATE`
- **EPI:** `epi_enrollments.next_visit_date DATE` — set to the nearest upcoming dose date across all vaccine types; service layer recomputes after each vaccination record
- **Nutrition:** no `next_visit_date` column — OPT+ visits are driven by monthly weighing sessions, not per-patient scheduling

### Nutrition Z-scores and classification
- `nutrition_visits` stores computed values: `waz NUMERIC`, `haz NUMERIC`, `whz NUMERIC`
- `nutrition_visits` also stores `nutrition_status TEXT` (e.g., `'NORMAL'`, `'UNDERWEIGHT'`, `'STUNTED'`, `'WASTED'`, `'SEVERELY_WASTED'`)
- Both Z-scores and classification are computed and set **on save** by the service layer — not recomputed at read time
- `severe_wasting BOOLEAN NOT NULL DEFAULT FALSE` remains as the ML trigger flag (set when WHZ < −3)

### ML integration fields (must be present for Phase 8)
- `prenatal_enrollments.is_high_risk BOOLEAN NOT NULL DEFAULT FALSE` — fed to at-risk classifier
- `epi_enrollments.fic_status BOOLEAN NOT NULL DEFAULT FALSE` — fed to barangay risk index
- `nutrition_visits.severe_wasting BOOLEAN NOT NULL DEFAULT FALSE` — WHZ < −3 flag, fed to at-risk classifier

### EPI vaccine column
- `epi_vaccinations.vaccine` — use **TEXT** (not enum) to avoid migration churn if DOH adds vaccines; validate allowed values at the Pydantic schema layer instead

### Pydantic schema structure
- Split by program: `schemas/prenatal.py`, `schemas/postpartum.py`, `schemas/epi.py`, `schemas/nutrition.py`
- Each file has `XxxCreate` + `XxxResponse` classes (16 total across 4 files)
- No shared base schema layer — follow the Phase 3 consultation pattern

### Alembic migration
- Single migration file covering all 8 tables + `record_status` shared enum — do not split by program
- Migration file: `alembic/versions/XXXX_mch_schema.py`
- Define `record_status` ENUM before the first table that uses it; subsequent tables reference the already-created type

### Claude's Discretion
- Exact column names and nullable constraints beyond the fields listed above
- Index strategy (beyond obvious PK/FK indexes)
- Whether to use PostgreSQL-level CHECK constraints or Pydantic-only validation for enums

</decisions>

- `TimestampMixin` + `SoftDeleteMixin` (`backend/app/core/base.py`): Apply to all 8 new tables

- `backend/app/models/health_station.py`: FK target — `health_stations.id`

### Established Patterns
- Soft deletes: `deleted_at TIMESTAMPTZ` column via `SoftDeleteMixin` — never hard DELETE
- `lazy="raise"` on all relationships — enforced project-wide
- Async-first: all SQLAlchemy operations use async sessions
- Schema files split by domain (e.g., `schemas/consultation.py`) — follow same split for MCH
- `health_station_id` excluded from create request bodies — auto-set from `current_user.health_station_id` in service layer (established Phase 3)
- `recorded_by` excluded from create request bodies — auto-set from `current_user.id` in service layer

### Integration Points
- `prenatal_enrollments.is_high_risk` → Phase 8 ML at-risk classifier
- `epi_enrollments.fic_status` → Phase 8 barangay risk index
- `nutrition_visits.severe_wasting` → Phase 8 ML at-risk classifier
- `record_status` shared ENUM → offline sync pipeline (Phase 9 PWA/BHW)
- `prenatal_enrollments.next_visit_date` + `postpartum_enrollments.day1/week1/week6_date` + `epi_enrollments.next_visit_date` → overdue detection queries in Phases 4.2 and 4.3

<specifics>
## Specific Ideas

- Single migration for all 8 tables — avoid ordering dependencies between multiple migration files
- `epi_vaccinations.vaccine` as TEXT resolved: validate in Pydantic with `Literal[...]` or a string enum, not a PostgreSQL ENUM type
- `record_status` shared ENUM defined once in this migration so TB/NCD/SURV tables in later phases can reference it without a new migration
- Z-scores and `nutrition_status` are denormalized into `nutrition_visits` intentionally — historical trend queries and ML queries need them readable without recomputation
- Postpartum `day1_date`/`week1_date`/`week6_date` are computed at enrollment creation from `delivery_date` — the service layer owns this calculation, not the ORM
- EPI `next_visit_date` = earliest upcoming dose across all vaccine types; service layer recomputes after every `epi_vaccinations` insert

</specifics>
