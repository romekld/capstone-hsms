# Plan 03-05: Patient Search Page (`/patients`)

## Objective
Implement the `/patients` route: a searchable, sortable, paginated patient list with a BHS-scope toggle, "Register Patient" primary action, skeleton loading, empty state, and error state per the UI-SPEC.

## Context
- Phase: 03 - Patient ITR + Core Data Model
- Depends on: 03-04 (types + API functions must exist; route must be registered in App.tsx)
- Blocked by: Nothing beyond 03-04

## Tasks

### Task 1: Build the patients list page

**Files to create/modify:**
- `frontend/src/pages/patients/PatientsPage.tsx` — the `/patients` route component

**Implementation notes:**

**Layout and structure (from UI-SPEC §Route: `/patients`):**
- Page container: `max-w-screen-xl mx-auto px-6 py-8 space-y-6` — consistent with admin pages
- Page h1: `"Patients"` — Display/28px (`text-2xl font-semibold`) — exactly one per route
- Top toolbar row: flex row, `items-center gap-4`:
  - Search `Input`: `placeholder="Search by name or patient ID"`, `className="max-w-[400px]"` on desktop, full-width on mobile. Bind to `searchQuery` state. Debounce API call by 300ms using `useEffect` + `setTimeout` (do not call on every keystroke)
  - "Search all BHS" `Checkbox` + `Label`: inline, hidden for CROSS_BHS_ROLES (CHO, DSO, PHIS coordinator — those roles always see city-wide). Determine user role from JWT/auth context
  - "Register Patient" `Button` variant `"default"` (primary): `className="ml-auto"`, navigates to `/patients/new` on click
- Results `Table`:
  - Columns: Patient Name | Birthdate | Sex | BHS | Registered — matching UI-SPEC
  - "Patient Name" displays `{last_name}, {first_name} {middle_name || ""}` — last name first for clinical context
  - "BHS" column shows a `Badge` using inline style `color: var(--bhs-tier)` when `city_wide` is true or user is CROSS_BHS_ROLES; suppress BHS column (or show as `—`) when scoped to own BHS
  - If `patient.possible_duplicate` is true, show a small `Badge` variant critical next to name with label `"Duplicate"` using `color: var(--status-critical)`
  - `TableRow` must have `onClick={() => navigate('/patients/' + patient.id)}` and `className="cursor-pointer hover:bg-muted/50"` (UI-SPEC interaction states: Table Row hover)
  - Row height minimum 44px (touch target rule — field nurses use tablets)
  - All columns are sortable: clicking a column header toggles sort direction. Manage `sortBy` and `sortDir` state, pass as query params to `listPatients()` (add `sort_by` and `sort_dir` query params to `PatientSearchParams` type if not already there, and handle in `PatientsPage` by client-side sorting the fetched array if backend does not yet support sort params)
  - Sortable column header: `aria-label="Sort by Patient Name"` on the sort button (UI-SPEC accessibility)
- Loading state: render 5 `Skeleton` rows matching table column widths (`className="h-10 w-full"`) while `isLoading` is true
- Empty state (no data, no active search): centered `<div>` with heading `"No patients registered yet"` and body `"Register your first patient to get started."` (exact copy from UI-SPEC §Copywriting Contract)
- Empty state (search returned no results): heading `"No patients found"`, body `"Try a different name or patient ID. Use \"Search all BHS\" to search across all health stations."` (exact copy from UI-SPEC)
- Error state: `sonner` `toast.error("Could not load patients. Check your connection and try again.")` — trigger in the catch block of the fetch; do not render an error component inline
- `Pagination`: shadcn `Pagination` component, 20 rows per page default (UI-SPEC). Track `page` state, compute `offset = (page - 1) * 20`. Show pagination only if total count > 20 — use the count endpoint from `PatientRepository` (add `GET /patients/count` route in 03-02 if not already done, or include `total` in the list response as a header `X-Total-Count`)

**State management:**
- `searchQuery: string` — controlled input value
- `cityWide: boolean` — scope toggle state; default false for nurse/midwife/physician; always true for CROSS_BHS_ROLES
- `patients: Patient[]` — fetched results
- `isLoading: boolean`
- `sortBy: keyof Patient` — column being sorted; default `"last_name"`
- `sortDir: "asc" | "desc"` — default `"asc"`
- `page: number` — current page; default 1

**Role detection for scope toggle visibility:**
- Read current user roles from auth context (look at how `UsersPage.tsx` reads auth state — follow the same pattern)
- `const isCrossBhsRole = currentUser?.roles.some(r => ["city_health_officer", "disease_surveillance_officer", "phis_coordinator"].includes(r))`
- If `isCrossBhsRole`, hide the "Search all BHS" checkbox and force `cityWide = true` in all API calls

**CSS variable usage:**
- `var(--bhs-tier)` for BHS badge color — use inline `style={{ color: "var(--bhs-tier)" }}` or a `className` with a CSS utility that references the variable. Never hardcode hex (UI-SPEC §Color)
- `var(--status-critical)` for possible_duplicate badge
- `var(--primary)` for "Register Patient" button (default shadcn Button variant already uses this)

**Animation:**
- Table skeleton → content: opacity fade 150ms (UI-SPEC §Animation Contract). Use `className={isLoading ? "opacity-0" : "opacity-100 transition-opacity duration-150"}`

### Task 2: Create `features/patients/components/PatientFormSections.tsx` stubs

**Files to create/modify:**
- `frontend/src/features/patients/components/PatientFormSections.tsx` — shared form section components used by RegisterPatientPage (03-05) and reused by future admin tooling

**Implementation notes:**
- Follow the structure of `frontend/src/features/admin/components/UserFormSections.tsx`
- Export these section components (used by RegisterPatientPage in Plan 03-05):
  - `IdentitySection` — Last Name, First Name, Middle Name fields
  - `DemographicsSection` — Birthdate, Sex fields
  - `AddressSection` — Barangay Select, Address Line field
  - `ContactSection` — Mobile Number field
  - `AssignmentSection` — Health Station Select
- Each section is a `React.FC` accepting `value` + `onChange` props for its fields. Follow the `FieldSet` + `FieldLegend` + `FieldLabel` + `FieldError` component pattern from `frontend/src/components/ui/field.tsx` (UI-SPEC §Component Inventory)
- Sections use `aria-required="true"` on required inputs and `aria-invalid="true"` + `aria-describedby` on fields with validation errors (UI-SPEC §Accessibility)
- Barangay Select and Health Station Select require data props: accept `barangays: {psgc_code: string, name: string}[]` and `healthStations: {id: number, name: string}[]` as props — fetching is the responsibility of the page, not the section component
- Minimum input height 44px on all inputs (`className="h-11"` or equivalent — touch target rule)
- All `Input` and `Select` components use focus ring: `ring-2 ring-ring ring-offset-2` on focus (handled by shadcn defaults if using the project's shadcn components; verify it works)
