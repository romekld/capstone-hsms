# Plan 03-06: Register Patient Page (`/patients/new`)

## Objective
Implement the `/patients/new` dedicated registration page with all required form sections, client-side validation, duplicate detection warning card with "Use Existing Patient" and "Register Anyway" actions, and audit-logged override flow.

## Context
- Phase: 03 - Patient ITR + Core Data Model
- Depends on: 03-04 (types + API), 03-05 (`PatientFormSections` shared components)
- Blocked by: Nothing beyond 03-04 and 03-05

## Tasks

### Task 1: Build the RegisterPatientPage with form and validation

**Files to create/modify:**
- `frontend/src/pages/patients/RegisterPatientPage.tsx` — the `/patients/new` route component

**Implementation notes:**

**Page structure (from UI-SPEC §Route: `/patients/new`):**
- Page entrance animation: add `className="login-page-enter"` to the outermost container (the class is already defined in `globals.css` — reuse it per UI-SPEC §Animation Contract)
- Page container: `max-w-2xl mx-auto px-6 py-8 space-y-8` — matches `CreateUserPage` pattern (RESEARCH.md §Pattern 10)
- Breadcrumb at top: shadcn `Breadcrumb` → `Patients > Register Patient`. "Patients" link navigates to `/patients`
- Page h1: `"Register Patient"` — Display/28px (`text-2xl font-semibold`)
- Form layout: single column on mobile; two-column grid on tablet+ (`md:grid md:grid-cols-2 md:gap-6`) — UI-SPEC §Route: `/patients/new`
- Form sections using `PatientFormSections` from 03-05 Task 2:
  1. `IdentitySection` — Last Name (required `*`), First Name (required `*`), Middle Name (optional)
  2. `DemographicsSection` — Birthdate (required `*`, `Input type="date"`), Sex (required `*`, `Select`)
  3. `AddressSection` — Barangay (required `*`, `Select` from barangay list), Address Line (optional)
  4. `ContactSection` — Mobile Number (optional)
  5. `AssignmentSection` — Health Station (required `*`, `Select` — pre-filled to nurse's `health_station_id` from JWT)
- Required field legend: text `"* Required"` above the submit area (UI-SPEC)
- First Name input receives `autoFocus` on page load — UI-SPEC says "First Name / first_name input — receives autofocus on page load" for the focal point
- **Barangay and Health Station data**: fetch from existing API endpoints. Check `frontend/src/features/admin/healthStations.ts` — if barangay list is already available as a static import or API call, reuse the same approach. Otherwise, call `GET /api/health-stations` and `GET /api/barangays` (check if these endpoints exist in the Phase 1/2 backend; they likely do since the admin Create User form has a Health Station selector)

**Form state:**
```typescript
const [form, setForm] = useState<PatientCreate>({
  last_name: "",
  first_name: "",
  middle_name: undefined,
  birthdate: "",
  sex: "male",  // default; user must explicitly choose
  barangay_psgc_code: "",
  address_line: undefined,
  health_station_id: currentUser?.health_station_id ?? 0,  // pre-fill from JWT
  mobile_number: undefined,
  force_create: false,
});
const [fieldErrors, setFieldErrors] = useState<Partial<Record<keyof PatientCreate, string>>>({});
const [isLoading, setIsLoading] = useState(false);
const [duplicates, setDuplicates] = useState<DuplicateMatch[] | null>(null);
const [forceCreate, setForceCreate] = useState(false);
```

**Client-side validation (run before API call):**
- `last_name`: non-empty → `"Last Name is required."` (exact copy from UI-SPEC §Copywriting: "Error state — Required field missing: {Field name} is required.")
- `first_name`: non-empty → `"First Name is required."`
- `birthdate`: non-empty and valid date → `"Birthdate is required."`
- `sex`: must be `"male"` or `"female"` → `"Sex is required."`
- `barangay_psgc_code`: non-empty → `"Barangay is required."`
- `health_station_id`: non-zero → `"Health Station is required."`
- Display errors inline under each field using `FieldError` component with `aria-invalid="true"` and `aria-describedby` linking input to error (UI-SPEC §Accessibility)

**Submit flow (`handleSubmit`):**
1. Run client-side validation; stop if any errors
2. `setIsLoading(true)` — disables submit button with spinner (UI-SPEC §Interaction States: Primary Button loading)
3. Try `createPatient({ ...form, force_create: forceCreate })`
4. On 201 success: `toast.success("Patient registered.")`, `navigate("/patients")`
5. On 409 response: extract `duplicates` from `error.response.data.detail.duplicates`; call `setDuplicates(duplicates)` to show the warning card; do NOT navigate
6. On other error: `toast.error("Could not save patient record. Check your connection and try again.")` (exact copy from UI-SPEC)
7. `finally { setIsLoading(false) }`

**Sticky footer action bar** (from RESEARCH.md §Pattern 10 — matches CreateUserPage):
- `className="sticky bottom-0 bg-background border-t border-border pt-4 pb-6 flex gap-3 justify-end"`
- "Back to Patients" `Button` variant `"secondary"` — navigates to `/patients`
- "Register Patient" `Button` variant `"default"` (primary) — `disabled={isLoading}`, shows spinner when loading
- Full-width on mobile, auto-width on desktop

### Task 2: Implement duplicate warning card

**Files to create/modify:**
- `frontend/src/pages/patients/RegisterPatientPage.tsx` — add duplicate warning section (same file as Task 1)

**Implementation notes:**
- Duplicate warning renders when `duplicates !== null && duplicates.length > 0`
- Position: inline above the sticky footer action bar — NOT a blocking modal (UI-SPEC: "Display as Alert component inline above form submit area")
- Use shadcn `Alert` component with a critical variant:
  ```tsx
  <Alert style={{ borderColor: "var(--status-critical)" }}>
    <AlertCircle style={{ color: "var(--status-critical)" }} />
    <AlertTitle>"Patient may already exist"</AlertTitle>
    <AlertDescription>
      {duplicates[0].last_name}, {duplicates[0].first_name}
      {/* comma-separated middle name if present */},
      registered at {duplicates[0].bhs_name ?? "Unknown BHS"} on {formatDate(duplicates[0].created_at)}.
      Review the existing record before creating a new one.
    </AlertDescription>
  </Alert>
  ```
  Exact copy from UI-SPEC §Copywriting: `"[Full Name], registered at [BHS Name] on [Date]. Review the existing record before creating a new one."`
- Animate card in with fade-down 200ms (UI-SPEC §Animation Contract). Wrap in a `div` with `className="animate-in fade-in slide-in-from-top-2 duration-200"` using `tw-animate-css` utilities (check globals.css for the available animation classes)
- Two action buttons inside the warning card (UI-SPEC §Duplicate Warning State):
  1. **"Use Existing Patient"** — `Button` variant `"outline"` — navigates to `/patients/${duplicates[0].id}`. This action cancels the new registration.
  2. **"Register Anyway"** — `Button` variant `"ghost"` with `style={{ color: "var(--status-critical)" }}` — sets `forceCreate = true` then re-calls `handleSubmit()`. Must be visually subordinate to "Use Existing Patient" (ghost vs outline — outline is more prominent)
- Below "Register Anyway": inline warning text `"This will be flagged for review by the administrator."` (exact copy from UI-SPEC §Copywriting). Style with `className="text-xs"` and `style={{ color: "var(--status-critical)" }}`
- Color is never the sole indicator: duplicate warning uses text heading + body + icon + color (UI-SPEC §Accessibility)
- "Register Anyway" flow: `force_create: true` is sent in the POST body → backend saves with `possible_duplicate=true` and writes audit log → frontend receives 201 and navigates to `/patients`
