**Create `frontend/src/pages/patients/RegisterPatientPage.tsx`:**

Build a dedicated registration page following UI-SPEC exactly:

1. **Page structure:**
   - Breadcrumb: `Patients > Register Patient` using shadcn Breadcrumb. "Patients" links to `/patients`.
   - Page h1 (Display/28px): "Register Patient" (text-[1.75rem] font-semibold)
   - Page entrance animation: `login-page-enter` class from globals.css

2. **Form layout:**
   - Single column on mobile; two-column grid on tablet+ (768px: `grid grid-cols-1 md:grid-cols-2 gap-x-8 gap-y-4`)
   - Use `FieldSet` + `FieldLegend` from field.tsx for each section
   - Use `Field`, `FieldLabel`, `FieldError` for each input (shadcn pattern, NOT raw divs)

3. **Form sections with exact fields:**

   **Section 1 - Identity (FieldSet):**
   - Last Name*: `Input` type="text", required, `aria-required="true"`
   - First Name*: `Input` type="text", required, autofocus on page load, `aria-required="true"`
   - Middle Name: `Input` type="text", optional

   **Section 2 - Demographics (FieldSet):**
   - Birthdate*: `Input` type="date", required, `aria-required="true"`
   - Sex*: `Select` with options Male / Female, required, `aria-required="true"`

   **Section 3 - Address (FieldSet):**
   - Barangay*: `Select` populated with BARANGAY_OPTIONS (hardcoded list of 32 barangays with psgc_codes — same pattern as HEALTH_STATIONS in admin). Required, `aria-required="true"`.
   - Address Line: `Input` type="text", optional, placeholder "House/Lot/Street"

   **Section 4 - Contact (FieldSet):**
   - Mobile Number: `Input` type="tel", optional, `inputmode="tel"`

   **Section 5 - Assignment (FieldSet):**
   - Health Station*: `Select` populated with HEALTH_STATIONS (reuse from admin types or define locally). Pre-filled to `user.health_station_id`. Required.
   - For non-admin roles, this field could be read-only/disabled since health_station_id is auto-set by backend from current_user. Display it for transparency but the backend ignores the value.

4. **Required field indicator:** Asterisk (*) next to FieldLabel for required fields. Legend note "* Required" above submit area.

5. **Client-side validation:**
   - Required fields: last_name, first_name, birthdate, sex, barangay_psgc_code
   - On submit, check all required fields are filled. Show `FieldError` with "{Field name} is required." inline under empty required fields.
   - Use `data-invalid` on Field and `aria-invalid` on control for validation errors (shadcn pattern)

6. **Submit flow:**
   - On submit, first call `registerPatient(body)` with `force_duplicate: false`
   - If backend returns 409 (duplicate found), show duplicate warning (see below)
   - If backend returns 201, show success toast and navigate to `/patients/${newPatient.id}`
   - If backend returns other error, show error toast: "Could not save patient record. Check your connection and try again."
   - Submit button: "Register Patient" (primary), full-width on mobile. Disabled during async with Spinner icon.

7. **Duplicate warning state:**
   - Triggered when backend returns 409 OR when a separate `checkDuplicate` call finds a match
   - Display as `Alert` component (variant destructive) inline ABOVE the submit button area — NOT a modal
   - Animate in with fade-down (tw-animate-css)
   - Warning heading: "Patient may already exist"
   - Warning body: "[Full Name], registered at [BHS Name] on [Date]. Review the existing record before creating a new one."
   - Two action buttons inside the alert:
     1. "Use Existing Patient" — `Button` variant="outline" — navigates to `/patients/${existingPatient.id}`
     2. "Register Anyway" — `Button` variant="ghost" with destructive text color (text-[hsl(var(--status-critical))]) — calls `registerPatient(body)` with `force_duplicate: true`
   - "Register Anyway" is visually subordinate to "Use Existing Patient"
   - Below "Register Anyway" button: inline text "This will be flagged for review by the administrator." in muted text

8. **Cancel button:** "Back to Patients" secondary Button, navigates to `/patients`

9. **Barangay and Health Station data:**
   Define as local constants (same pattern as HEALTH_STATIONS in admin). For the 32 barangays, create a `BARANGAY_OPTIONS` array with `{ psgc_code: string, name: string }`. The psgc_codes MUST match what was seeded in migration 0002 — the executor MUST read `backend/fixtures/cho2-boundaries.geojson` (or the seed migration `backend/alembic/versions/0002_seed_barangays_and_health_stations.py`) to extract the exact psgc_codes and names. Do NOT use placeholder codes — they will cause FK constraint violations on patient registration.

   Similarly, reuse or duplicate the HEALTH_STATIONS constant from the admin module (32 BHS names, IDs 1-32).

10. **Responsive:** Single column on mobile, two columns on md+ for form fields.
